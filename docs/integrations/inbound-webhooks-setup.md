# Inbound Webhooks Setup — Resend & Gupshup

What you need to configure outside the code so the Conversation Agent and Morning Brief actually work end-to-end. Code is shipped; this is the ops side.

## TL;DR

| What | Where | Value |
| --- | --- | --- |
| Resend inbound webhook | Resend dashboard → Inbound | `https://<your-convex-site-url>/email/inbound` (POST) |
| Resend inbound domain (MX) | DNS for `rolerecruit.com` | Resend-provided MX records on `reply.rolerecruit.com` (recommended subdomain) |
| Gupshup inbound webhook | Gupshup dashboard → App → Callback URL | `https://<your-convex-site-url>/whatsapp/inbound` (POST) |
| `REPLY_INBOUND_DOMAIN` | Convex env | `reply.rolerecruit.com` (matches the subdomain whose MX points at Resend) |
| `PUBLIC_BASE_URL` | Convex env | `https://app.rolerecruit.com` (used in reschedule booking links) |
| `RESEND_API_KEY` | Convex env | from Resend dashboard |
| `GUPSHUP_API_KEY`, `GUPSHUP_APP_NAME`, `GUPSHUP_SOURCE_NUMBER` | Convex env | from Gupshup dashboard |

Your Convex site URL is the value of `NEXT_PUBLIC_CONVEX_SITE_URL` (or the `.convex.site` domain shown in the Convex dashboard). HTTP routes in [convex/http.ts](convex/http.ts) are served at that origin, not the Convex API URL.

## 1. Resend (email replies)

### 1a. Inbound domain

Resend needs to receive replies on a domain whose MX records point at Resend. The webhook handler at [convex/http.ts:10](convex/http.ts:10) accepts any inbound JSON Resend sends; the routing happens via the `to:` address.

Recommended setup: add a subdomain (e.g. `reply.rolerecruit.com`) in Resend → Domains → Add Domain → Receiving. Resend will print MX records — add them at your DNS provider. The apex domain can keep sending via SPF/DKIM as before; receiving lives on the subdomain.

### 1b. Webhook endpoint

In Resend dashboard → Inbound:

- **Endpoint URL**: `https://<convex-site-url>/email/inbound`
- **Method**: POST
- **Payload**: default Resend inbound JSON

The handler expects this shape (see [convex/http.ts:14-30](convex/http.ts:14)):

```json
{
  "to": "reply+abc123...@reply.rolerecruit.com",
  "from": "candidate@example.com",
  "subject": "...",
  "text": "...",
  "html": "...",
  "attachments": []
}
```

`to` is the only field used for routing. The router at [convex/email_reply_router.ts:5](convex/email_reply_router.ts:5) extracts a 32-char lowercase alphanumeric token via `/^reply\+([a-z0-9]{32})@/i`. If a matching `outreachMessages` row exists by `replyToken`, the reply is threaded into that conversation; otherwise it falls through to the careers-portal resume ingestion path.

### 1c. Reply-To wiring (now in code)

Both outreach emails (triage agent) and conversation-agent replies set `Reply-To` automatically. The dispatcher at [convex/outreach.ts:298](convex/outreach.ts:298) reads `msg.replyToken` (generated when the draft is created at [convex/outreach.ts:154](convex/outreach.ts:154)) and composes `reply+<token>@${REPLY_INBOUND_DOMAIN}`. Agent replies generated by the conversation agent at [convex/conversation.ts:81](convex/conversation.ts:81) also get their own token so a candidate replying to an auto-reply still threads correctly.

Set the inbound subdomain to match whatever you configured in Resend (step 1a):

```bash
bunx convex env set REPLY_INBOUND_DOMAIN reply.rolerecruit.com
```

Default if unset: `reply.rolerecruit.com`. WhatsApp does not need an analogous env var — matching is phone-based, see step 2c.

### 1d. Verify

After DNS propagation and webhook config, send a test:

```bash
curl -X POST "https://<convex-site-url>/email/inbound" \
  -H 'Content-Type: application/json' \
  -d '{
    "to":"reply+aaaaaaaaaaaaaaaaaaaaaaaaaaaaaaaa@reply.rolerecruit.com",
    "from":"candidate@example.com",
    "text":"What are the school timings?"
  }'
```

Expected: `200 {"success":true,"routed":"new_resume"}` (because that token won't exist yet). Then send a real outreach email through the app, copy its `replyToken` from the Convex dashboard, and reply with that token — you should see a `candidate_reply` row appear in `outreachMessages` and either an auto-reply (FAQ/reschedule) or an escalation to the Inbox.

## 2. Gupshup (WhatsApp replies)

### 2a. Webhook endpoint

In Gupshup dashboard → App → Settings → Callback URL:

- **URL**: `https://<convex-site-url>/whatsapp/inbound`
- **Method**: POST
- **Version**: v3 (recommended)

### 2b. Payload shape

The handler at [convex/http.ts:32-51](convex/http.ts:32) expects:

```json
{
  "type": "message",
  "payload": {
    "source": "+919876543210",
    "payload": { "text": "Can we reschedule?" }
  }
}
```

It accepts the older `payload.text` variant too. Anything without both `source` and `text` is silently ignored (200 OK, `ignored: true`).

### 2c. How matching works

WhatsApp has no reply token to thread on, so the matcher in [convex/whatsapp.ts:107-135](convex/whatsapp.ts:107) does this:

1. Normalize the `source` phone to E.164 via [convex/lib/phone.ts](convex/lib/phone.ts) (libphonenumber-js, default country `IN`).
2. Find a `candidates` row whose phone normalizes to the same E.164 string.
3. Find the most recent outbound message to that candidate that is (a) not type `rejection`, (b) sent within the last 30 days.
4. Use that message's `applicationId` + `schoolId` to thread the reply.

Phones are normalized to E.164 at every write site (careers form, manual creation, AI resume parsing, email ingestion), so stored values look like `+919876543210`. Gupshup also delivers E.164, so matching is just a string equality check after both sides go through the normalizer. The old phone-format gotcha is gone — bare 10-digit numbers, dashes, spaces, and parens all reduce to the same canonical form.

#### Backfill existing rows

Rows that pre-date the normalization change may still have heterogeneous phone formats. One-off mutation to clean them up:

```bash
bunx convex run candidates:backfillNormalizePhones
```

Returns `{ normalized, unchanged, invalid, skipped }`. `invalid` rows are left untouched so you can inspect and fix them manually — the matcher will skip them until then.

### 2d. Verify

```bash
curl -X POST "https://<convex-site-url>/whatsapp/inbound" \
  -H 'Content-Type: application/json' \
  -d '{
    "type":"message",
    "payload":{
      "source":"+919876543210",
      "payload":{"text":"What are the timings?"}
    }
  }'
```

Expected: `200 {"success":true,"matched":true,"applicationId":"..."}` if that phone has a recent outbound, or `{"matched":false}` if not.

## 3. Convex env vars

Set on the Convex deployment (`bunx convex env set`), not just in `.env.local`:

```bash
bunx convex env set RESEND_API_KEY <key>
bunx convex env set REPLY_INBOUND_DOMAIN reply.rolerecruit.com
bunx convex env set GUPSHUP_API_KEY <key>
bunx convex env set GUPSHUP_APP_NAME <app-name>
bunx convex env set GUPSHUP_SOURCE_NUMBER <e164-number>
bunx convex env set PUBLIC_BASE_URL https://app.rolerecruit.com
bunx convex env set GOOGLE_API_KEY <key>
```

`PUBLIC_BASE_URL` is referenced at [convex/conversation.ts:140](convex/conversation.ts:140) when the agent generates a reschedule booking link. Without it, links default to `https://rolerecruit.com/book/<token>` which may not be your environment.

`GOOGLE_API_KEY` is what the conversation agent uses for classification and FAQ drafting via the OpenAI-compatible Gemini endpoint. Without it, the agent escalates everything.

## 4. Per-school feature flags

Both features are opt-in per school. After webhooks are wired up, turn them on in the web app (no Convex dashboard needed):

- **Conversation agent**: Settings → Conversation agent → toggle "Enable agent" + paste FAQ content.
- **Morning brief**: Settings → Notifications → toggle "Send morning brief" + check the recipients you want.

When `conversationAgentEnabled` is `false` (default), every inbound reply lands in the Inbox tab for a human to handle — the webhook still fires, the message is still ingested, but the agent skips classification and auto-escalates with reason `agent_disabled`.

When `morningBriefEnabled` is `false` (default) or `morningBriefRecipientUserIds` is empty, the 02:30 UTC cron skips that school entirely.

## 5. Ordering checklist

A clean rollout looks like this:

1. Set Convex env vars including `REPLY_INBOUND_DOMAIN` (step 3).
2. Add MX records for the inbound subdomain (step 1a). Wait for DNS to propagate — `dig MX reply.rolerecruit.com` should return Resend's hosts.
3. Configure Resend inbound webhook (step 1b).
4. Configure Gupshup webhook (step 2a) and spot-check phone-number formats in `candidates` (step 2c).
5. Run `bunx convex run candidates:backfillNormalizePhones` once to normalize legacy phone formats. Note the `invalid` count and fix those rows by hand if needed.
6. Turn on the per-school flags for one pilot school (step 4).
7. Run the curl smoke tests (1d, 2d) against staging or prod.
8. Send a real outreach to a test candidate inbox you control. Reply. Confirm the message lands in `outreachMessages`, the agent either auto-replies or escalates, and the Inbox tab shows it if escalated.

## 6. Troubleshooting

**Inbound webhook fires but `routed: "new_resume"` always returned.** Resend probably stripped the `reply+<token>@...` Reply-To because the inbound subdomain isn't owned by your account, OR `REPLY_INBOUND_DOMAIN` is unset while the MX record lives on a different host. Check `outreachMessages.replyToken` exists in the dashboard, then compare the `to:` Resend posts to your webhook against `reply+<that-token>@<the-domain-you-configured>`.

**WhatsApp inbound returns `matched: false` for known candidates.** Either the stored phone failed E.164 normalization (check by running `bunx convex run candidates:backfillNormalizePhones` — the `invalid` count tells you how many rows need manual cleanup), or the candidate has no recent (≤30 day) outbound message that wasn't a rejection.

**Agent escalates everything even when enabled.** `GOOGLE_API_KEY` is missing or the LLM call is throwing. Check Convex logs for `[conversation_classify]` errors — the classifier falls back to `unclear` (which escalates) on any error.

**Reschedule link is broken.** `PUBLIC_BASE_URL` not set; agent defaults to `https://rolerecruit.com/book/<token>` which may not resolve.

**Morning brief never sends.** Check three things in order: (1) `morningBriefEnabled` is true for the school, (2) `morningBriefRecipientUserIds` has at least one entry, (3) `RESEND_API_KEY` is set on the Convex deployment.
